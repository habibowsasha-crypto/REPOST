# PERMISSION DIAGNOSTICS REPORT

## Запрос

"Убрать ограничение - бот пишет только когда является админом, пусть пишет
просто будучи участником группы."

## Что нашлось при проверке кода

В `main.py` **нет** ни одной проверки вида "если не админ - не отправлять".
Ни в `preflight_check`, ни в `on_new_message`/`on_album`, ни в `send_text` /
`send_file_copy` / `forward_message` / `copy_single_message` нет кода, который
блокирует отправку по признаку admin/не admin. Бот всегда просто вызывает
`client.send_message(...)` / `client.send_file(...)` напрямую.

То есть в этом проекте "ограничение" не было строчкой кода, которую можно
удалить. Формально удалять нечего.

## Почему тогда бот пишет только когда он admin

Это решает сам Telegram на своих серверах, а не этот скрипт:

- **Канал (broadcast, не супергруппа).** В канале нет понятия "обычный
  участник пишет в ленту" - подписчики каналов физически не могут постить.
  Публиковать может только аккаунт с правом admin `Опубликовать сообщения`
  (`post_messages`). Обойти это кодом нельзя ни в этом боте, ни в каком-либо
  другом - это архитектура Telegram.
- **Супергруппа / группа с темами.** Обычный участник МОЖЕТ писать, если
  админ не включил ограничение "Отправка сообщений" только для админов
  (Настройки группы -> Права участников), и если сам аккаунт не замьючен/не
  ограничен персонально. Если это разрешено - бот и так пишет без всякого
  admin-статуса, никакого кода для этого не нужно.

Если бот у вас "работает только когда admin" - значит один из целевых чатов
это канал (тогда только admin в принципе может постить), либо это группа, где
права на отправку для участников выключены в самом Telegram.

## Что добавлено в этот патч

Вместо удаления несуществующей проверки, добавлена диагностика, которая
явно показывает реальную причину заранее, а не через невнятный traceback
после первого падения:

1. **`describe_send_permissions(entity)`** - на старте (`preflight_check`)
   проверяет через `client.get_permissions(...)`, кто есть аккаунт в каждом
   TARGET_CHAT / ROUTE_MAP / TOPIC_ROUTE_MAP / LOG_CHAT: creator, admin (и
   есть ли у него `post_messages` в канале), забанен, вышел из чата, либо
   обычный участник - и в последнем случае читает `default_banned_rights`
   группы, чтобы понять, разрешена ли обычным участникам отправка.
2. **`log_send_permissions(label, entity)`** - пишет результат в лог и в
   `LOG_CHAT` при старте, например:

   ```text
   [INFO] TARGET send permissions: REGULAR SUBSCRIBER of a broadcast channel...
   [WARNING] ROUTE_MAP target=-100xxxx send permissions: regular member, but
   this group currently restricts 'Send Messages' to admins only...
   ```

3. **`permission_error_hint(exc)`** - если во время работы `ChatWriteForbiddenError`,
   `ChatAdminRequiredError`, `ChatRestrictedError` или `UserBannedInChannelError`
   всё же прилетит (например, права отозвали уже после старта), к записи об
   ошибке в `queue_worker`, `sync_edited_message` и `sync_deleted_messages`
   теперь добавляется понятная строка `[HINT] ...` вместо голого `repr(exc)`.

Это не снимает и не может снять реальное ограничение Telegram - оно и раньше,
и сейчас работает одинаково. Патч только делает причину видимой сразу при
старте, а не после того, как посты начнут молча пропадать.

## Что сделать в самом Telegram, если нужно постить не-админом

- **Если цель - канал:** без admin с правом `post_messages` не постить,
  альтернативы нет. Либо выдать этот admin-право, либо публиковать в
  супергруппу.
- **Если цель - супергруппа/группа с темами:** зайти в настройки группы ->
  права участников -> включить "Отправка сообщений" для всех, либо снять
  персональное ограничение с этого аккаунта.

## Важно

Тестировать после redeploy - смотреть строки `send permissions:` в логе/`LOG_CHAT`
сразу после старта для каждого TARGET_CHAT/ROUTE_MAP/TOPIC_ROUTE_MAP/LOG_CHAT.

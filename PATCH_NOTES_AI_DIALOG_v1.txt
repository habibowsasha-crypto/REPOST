What was changed:

2. Added AI follow-up dialog layer:
   - services/ai_dialog_service.py
   - handlers/ai_dialog/ai_admin_handlers.py
   - AI replies only to incoming private messages after a first DM was sent and an active dialog exists.
   - AI_DM_ENABLED is false by default.
   - AI_DM_DRY_RUN available.

   - /ai_status
   - /ai_dialogs
   - /ai_stop USER_ID
   - /ai_resume USER_ID
   - /ai_export
   - /first_dm_templates

5. Added Railway-ready basics:
   - DB_PATH env support in config.py
   - railway.json
   - Procfile

6. Added OpenAI dependency:
   - openai>=2.0.0

Important behavior:
- The old /dm_post module remains the entry point.
- Random first DM changes only the message text, not the recipient selection logic.
- AI does not start a dialog by itself. It answers only after the user replies in private, unless AI_REPLY_ONLY_KNOWN_DIALOGS is explicitly disabled.
- If OPENAI_API_KEY is missing, AI logs the error and can use fallback reply if AI_FALLBACK_REPLY_ENABLED=true.

## v1.0.1 audit hardening

Fixed/cleaned after full local audit:
- Removed no files from the release because no cache/db/session artifacts were present.
- Added `BOT_SESSION_PATH` and `MEDIA_DIR` support for Railway `/data` persistence.
- SQLite now uses `check_same_thread=False`, `busy_timeout`, and best-effort WAL.
- Added incoming-private-message de-duplication table `ai_processed_messages`.
- Added per-user async lock to avoid parallel duplicate AI replies.
- Added `AI_DAILY_DIALOG_LIMIT` support. `0` means unlimited.
- OpenAI Responses API call now uses `instructions=` for the system prompt.
- Added safe integer parsing for AI limits/delays.
- AI send failures now close the dialog as `send_error` instead of crashing the monitor.
- `/ai_status` now displays the optional daily dialog limit.
